# 判定ズレチェック

**① バイネーム競合チェック**
具体的な競合製品名・企業名がバイネームで出ている場合（例：キーエンス・UiPath・WinActor・Salesforce等）、「当て馬」「情報収集」リスクが高い。
→ 減点。競合がLIJより明らかに優位な領域なら見送り検討。

**② 予算×スコープ乖離チェック**
要件のスコープに対して予算が明らかに不足している場合（例：AI×RPA×システム連携×BIのフルスコープを500万円）。
→ スモールスタートに誘導できる余地があるかを確認。誘導不可ならC。

**③ 組織の重さチェック**
以下が重なる場合はタイムライン長期化・当て馬リスク大。

- 決裁フローが4段階以上
- 合併中・方針未確定・調査段階
- スケジュールがすべて「未定」
→ 2つ以上重なればB以下。3つ重なればC検討。

**④ 技術的地雷チェック**
一見AIっぽく見えても技術難易度が極めて高い要件が含まれる場合。

- 日中間・海外間のセキュアなデータ連携
- CAD（SolidWorks・iCAD等）との深い連携
- レガシー基幹システムとのAPI連携（仕様不明）
→ 難易度の高さをコメントに明記し、スコープ限定を前提にした場合のみエントリー可。

**⑤ AI案件かどうかの本質チェック**
「AI」「DX」というワードがあっても、本質がAI案件かを確認。

- データ統合管理案件ではないか
- SaaS導入代行ではないか
- システム設計・パターン設計案件ではないか
→ LIJの強みであるAI×業務効率化と本質的にズレている場合はC。
「AIエージェント」バズワード案件の識別チェック
クライアントが「AIエージェント」と記載していても、必要機能が未定・業務棚卸しが未実施の場合は、AIシステム化ではなく業務整理コンサルが先行するため、住吉さんのリソース活用前提の案件では特に優先度を下げること。

**⑥ 今動くべきタイミングチェック**
スコアが高くても「今エントリーすべきか」を判断する。

- 着手が1年以上先
- 「情報収集段階」「社内承認待ち」
- 選定時期・着手・納期がすべて未定
→ 種まきとしての価値があるかを評価。リソースが逼迫している場合はB〜C。

**⑦ LIJの現在地とのミスマッチチェック**
案件が求めるコンサルレベル・技術レベルがLIJの現在地を超えていないか。

- 組織変革管理・業務浸透まで求めるコンサルファームレベル
- 複数拠点・グループ会社横断の大規模グランドデザイン
- LIJに実績がないニッチな専門領域
→ 「経験値積み」として挑戦するか、見送るかをコメントで明記。

**⑧ LIJの得意な戦い場チェック**
LIJのフルスクラッチ開発・AI実装の強みが活きる案件か。

- HubSpot・Salesforce・SaaS専門会社が有利な案件ではないか
- 既存パッケージのアドオン開発で終わる案件ではないか
- LIJより明らかに適した専門会社が存在する案件ではないか
→ 該当する場合は「専門会社案件」としてC。

**⑨ 部門サイロ型発注チェック**
複数部門がバラバラに別ベンダーでAI・システムを入れようとしていないか。

- 「営業・総務は既存コンサル、技術領域はLIJに相談」のような分断はないか
- 全社のAI・データ戦略を誰も描いていない状態ではないか
- 部門別に入れた結果、データが連携されず活用の天井が低くなるリスクはないか
→ 該当する場合は「全社統合の壁打ち役」としてLIJが上位ポジションを取れるサイン。S転換の可能性あり。エントリー文面の訴求軸を「部分解決者」ではなく「全社データ戦略を整理できる唯一のパートナー」に変更する。
